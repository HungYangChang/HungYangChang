# Open-Source Contribution Cadence

> Option C: Authority Track — Sustained OSS engagement for long-term visibility.

---

## Goal

Maintain a visible, consistent open-source contribution pattern that signals:
1. Active engagement with the AI/ML community
2. Ability to read and contribute to production codebases
3. Collaborative engineering skills

## Cadence

| Activity | Frequency | Time Investment |
|----------|-----------|----------------|
| Own projects: maintenance + features | Weekly | 2–4 hrs/week |
| External contributions: issues + PRs | Bi-weekly | 2–3 hrs/session |
| Technical writing (cross-posted) | 2× per month | 3–5 hrs/article |

## Target Projects for Contributions

Focus on projects that align with the AI/LLM engineering brand:

### Tier 1: High-Alignment (Contribute Regularly)
- **LangChain / LangGraph** — LLM orchestration (direct skill alignment)
- **vLLM** — Inference optimization (connects to throughput work)
- **FastAPI** — Python backend framework (daily-use tool)
- **Pytest** — Testing framework (testing strategy expertise)

### Tier 2: Supporting (Contribute Opportunistically)
- **gRPC Python** — Microservices infrastructure
- **Ruff** — Python linting (developer tooling)
- **Hugging Face Transformers** — ML model ecosystem
- **ONNX Runtime** — Model deployment optimization

### Tier 3: Community Building
- Answer questions on GitHub Discussions in Tier 1 projects
- Write "good first issue" guides for own projects
- Review PRs from other contributors on own projects

## Contribution Types (by effort)

### Low Effort (Start Here)
- Fix typos in documentation
- Add type hints to untyped functions
- Improve error messages
- Add missing tests for edge cases

### Medium Effort
- Fix reported bugs with tests
- Add new examples or tutorials
- Improve CI/build workflows
- Write integration guides

### High Effort (Build To)
- Implement requested features
- Refactor modules for performance
- Design new components or APIs

## Monthly Rhythm

| Week | Focus |
|------|-------|
| Week 1 | Maintain own repos (merge PRs, close issues, update deps) |
| Week 2 | External contribution (pick one issue from Tier 1 project) |
| Week 3 | Publish technical article (cross-post to Dev.to / Medium) |
| Week 4 | External contribution or own-project feature work |

## Tracking

### GitHub Contribution Graph
Keep the contribution graph green with consistent activity:
- Minimum: 3–4 days/week with commits
- Mix: Own repos + external contributions + technical writing repo

### Monthly Review
At the end of each month, log:
- [ ] Number of external PRs submitted
- [ ] Number of external PRs merged
- [ ] Articles published
- [ ] New stars/forks on own repos
- [ ] Interesting interactions or connections made

## Getting Started: First 30 Days

1. **Week 1:** Set up this tracking system. Pick one Tier 1 project and read its CONTRIBUTING.md.
2. **Week 2:** Submit your first contribution (documentation fix or small bug fix).
3. **Week 3:** Publish your first technical article on Dev.to.
4. **Week 4:** Submit a second external contribution (slightly more substantial).

## Anti-Patterns to Avoid

- **Spray-and-pray PRs** — Don't submit low-quality PRs to many repos just for green squares
- **Abandoned own repos** — Don't neglect your own projects while contributing externally
- **All writing, no code** — Balance articles with actual code contributions
- **Invisible work** — Make sure contributions are public and linked to your profile
